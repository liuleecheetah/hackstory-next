# HackStory 時間軸檔案格式規格（`.hst.json`）

**版本：** 0.2（草案）
**狀態：** 開發中，尚未凍結
**最後更新：** 2026-07-13

> 版本歷史：0.2 新增 `events[].ongoing`（進行中事件）。依第 10 節規則，0.1 的檔案仍可正常讀取。

---

## 0. 這份文件在幹嘛

這是 HackStory 的**地基**。

一條時間軸不是「資料庫裡的一筆資料」，而是**一個可以被下載、寄送、備份、引用、重新混合的獨立檔案**。這個決定讓以下事情變成可能：

- 你做好一條軸，可以直接把檔案給別人
- 別人做好的軸，你可以載進來跟自己的疊在一起對比
- 共用資料庫壞掉／關站／被封鎖，時間軸本身還在
- 可以放進 GitHub 做版本控管，看得出誰在什麼時候改了哪一則事件

**因此：格式先於程式。** 這份規格改動的成本會隨時間急遽上升，所以現在要想清楚。

---

## 1. 核心概念與名詞

| 中文 | 英文 | 說明 |
|---|---|---|
| 時間軸文件 | Timeline Document | 一個 `.hst.json` 檔案。**發佈與分享的最小單位** |
| 軸線 | Track | 文件內的一條水平線。一個文件可有多條軸線（例：「立法進程」「街頭運動」） |
| 事件 | Event | 軸線上的一個點或一段區間 |
| 關係 | Relation | 兩個事件之間的連結（因果、回應、衍生⋯⋯） |
| 圖層 | Layer | **執行期**概念，不存在檔案裡。使用者在畫面上疊加的多個「時間軸文件」，每個文件是一個圖層 |

> **關鍵區分：** 「軸線（Track）」是**文件內部**的分軌；「圖層（Layer）」是**跨文件**的疊加。
> 你的核心需求「取用別人的時間軸同步呈現對比」＝ 多圖層。

## 2. 檔案結構總覽

```json
{
  "hackstory": "0.2",
  "id": "tw-marriage-equality-legislation",
  "meta": { ... },
  "tracks": [ ... ],
  "events": [ ... ],
  "relations": [ ... ],
  "display": { ... }
}
```

| 欄位 | 必要 | 說明 |
|---|---|---|
| `hackstory` | ✅ | 規格版本號。程式靠這個判斷怎麼讀 |
| `id` | ✅ | 全域識別碼。用 slug（英數與連字號），發佈到共用庫時用來引用 |
| `meta` | ✅ | 這條時間軸的身分證 |
| `tracks` | ✅ | 軸線清單，至少一條 |
| `events` | ✅ | 事件清單，可為空陣列 |
| `relations` | ❌ | 事件關係，可省略 |
| `display` | ❌ | 建議的呈現方式（顏色、預設時間尺度）。載入者可覆寫 |

## 3. `meta` — 時間軸的身分證

```json
"meta": {
  "title": "台灣同婚立法進程",
  "subtitle": "從祁家威到釋字 748",
  "description": "整理 1986–2017 年間同性婚姻在台灣的法制化過程。",
  "authors": [
    { "name": "劉李俊達", "url": "https://g0v.tw" }
  ],
  "license": "CC-BY-4.0",
  "language": "zh-TW",
  "topics": ["人權", "婚姻平權", "立法"],
  "created": "2016-12-17",
  "updated": "2026-07-11",
  "revision": 12,
  "source": {
    "type": "google-sheet",
    "url": "https://docs.google.com/spreadsheets/d/..."
  }
}
```

**設計理由：**

- `license` **不是可有可無的**。這個專案的資料多半來自公民社會的共同整理，若要進入共用資料庫被別人取用、混合、再發佈，授權必須從第一天就內建在檔案裡。預設 `CC-BY-4.0`。
- `authors` 是陣列 → 支援共筆。
- `revision` 是單調遞增的整數，Phase 2 做同步／合併時用得到。
- `source` 記錄原始出處，讓「這份資料哪來的」永遠可追溯。

## 4. `TimePoint` — 最重要、也最容易做錯的東西

現實世界的時間是**模糊的**。你的食安資料裡就有「2010年6月」，同婚資料裡有「1986/3/7」也有「2016/11/24 09:00-18:00」。有些事件甚至只知道先後、不知道日期。

如果只用一個 ISO 字串存時間，這些全部會被壓扁成假精確。所以：

```json
{
  "value": "2017-05-24",
  "precision": "day"
}
```

### `precision` 可為

| 值 | `value` 格式 | 範例 |
|---|---|---|
| `year` | `YYYY` | `"2010"` |
| `month` | `YYYY-MM` | `"2010-06"` ← 你資料裡的「2010年6月」 |
| `day` | `YYYY-MM-DD` | `"2017-05-24"` |
| `minute` | `YYYY-MM-DDTHH:mm` | `"2016-11-24T09:00"` |

### 選用欄位

```json
{
  "value": "2010-06",
  "precision": "month",
  "circa": true,          // 「大約」，畫面上會用虛化邊緣表示
  "raw": "2010年6月"       // 原始輸入字串，永遠保留，方便回溯與除錯
}
```

### 相對時間（**已實作**）

不知道確切日期、只知道順序時：

```json
"start": {
  "relative": {
    "after": "evt-003",
    "before": "evt-007"
  }
}
```

程式會把所有相對約束傳播收斂，推算出一個**合成位置**畫在軸上（虛線空心圓點），並在視覺上明確標示「這是推估位置，不是真實日期」。循環或互相矛盾的約束不會亂畫，會回報給使用者。

> **設計沿革：** MVP 先只實作絕對時間，但 `start` 欄位的型別從一開始就是「絕對時間 **或** 相對錨點」的聯集。
> **這是本規格最重要的前瞻設計，當初沒為了省事拿掉。**
> ——後記：相對時間的求解與繪製已於 0.2 時期實作（`core/relative.ts`），當初留的位置直接派上用場，一個欄位都沒改；畫面上也能用下拉選單設定「在 A 之後、B 之前」，不需手寫 JSON。

## 5. `tracks` — 軸線

```json
"tracks": [
  {
    "id": "legislation",
    "title": "立法與司法進程",
    "description": "國會、行政、司法系統內的動作",
    "color": "#3b6ea5",
    "order": 1
  },

    "id": "movement",
    "title": "社會運動",
    "color": "#d97706",
    "order": 2
  }
]
```

`id` 只需在**同一份文件內**唯一。

## 6. `events` — 事件

```json
{
  "id": "evt-018",
  "track": "legislation",
  "title": "司法院公布釋字第 748 號，宣告現行民法未保障同志婚姻違憲",
  "description": "大法官宣告民法第四編親屬第二章婚姻規定，未使同性別二人成立親密性與排他性之永久結合關係，與憲法第 22 條保障婚姻自由及第 7 條保障平等權之意旨有違，有關機關應於二年內完成修法。",
  "start": { "value": "2017-05-24", "precision": "day" },
  "end": null,
  "location": { "name": "司法院，台北" },
  "importance": 5,
  "confidence": "verified",
  "tags": ["釋憲", "大法官"],
  "sources": [
    { "title": "司法院釋字第 748 號解釋", "url": "https://cons.judicial.gov.tw/..." }
  ],
  "media": [],
  "color": null
}
```

### 欄位說明

| 欄位 | 必要 | 說明 |
|---|---|---|
| `id` | ✅ | 文件內唯一。**一旦發佈就不可改**（別人的關係線與評論會指向它） |
| `track` | ✅ | 對應 `tracks[].id` |
| `title` | ✅ | 一句話講完。這是軸上唯一會直接顯示的文字 |
| `description` | ❌ | 完整說明，點開才看得到 |
| `start` | ✅ | TimePoint |
| `end` | ❌ | TimePoint。有 `end` = 區間事件（畫成長條），無 = 點事件 |
| `ongoing` | ❌ | `true` = 事件仍在持續中：畫成從 `start` 延伸到「今天」的長條，右端以淡出表示未結束。與 `end` 擇一使用（兩者都有時以 `end` 為準）。0.2 新增 |
| `location` | ❌ | `{ name, lat?, lng? }`。留 `lat/lng` 位置給未來的地圖功能 |
| `importance` | ❌ | 1–5。`5` 視為「關鍵事件」，在軸上放大、加光暈、粗體顯示（介面上「標示為關鍵事件」即設為 5）。也保留給未來「縮小時哪些事件優先顯示」使用 |
| `confidence` | ❌ | `verified` / `reported` / `disputed` / `unknown` |
| `tags` | ❌ | 字串陣列 |
| `sources` | ❌ | `{ title, url }` 陣列 |
| `media` | ❌ | `{ type, url, caption }`，type 為 `image` / `video` / `document` |
| `color` | ❌ | 覆寫該事件顏色 |

### 為什麼要有 `confidence` 與 `sources`

這個工具的目的是「建立公共討論的共同認知基礎」。如果一則事件無法追溯來源、也不標示查證程度，它就無法承擔這個任務——反而會變成散播錯誤資訊的載體。

**`sources` 在 MVP 是選填，但介面上要主動提示；Phase 2 發佈到共用資料庫時，建議設為必填。**

## 7. `relations` — 事件關係

這是 HackStory 跟 TimelineJS 最大的差異：**時間軸要呈現的不只是「發生了什麼」，而是「為什麼會這樣」。**

```json
"relations": [
  {
    "from": "evt-015",
    "to": "evt-018",
    "type": "causes",
    "label": "祁家威釋憲聲請 → 大法官作成解釋"
  }
]
```

### `type` 允許值

| 值 | 意義 |
|---|---|
| `causes` | 導致 |
| `responds_to` | 回應／反制 |
| `derives_from` | 衍生自 |
| `contradicts` | 與之矛盾 |
| `same_event` | 同一事件的不同記載（**跨圖層對比時極為有用**：兩個人整理的軸中，這兩則其實是同一件事） |

> `same_event` 是把「圖層疊加」從視覺並列升級成真正的**知識整合**的關鍵。

**狀態：已實作。** 關係線會畫在事件之間，滑鼠移到事件上或點選就亮起並顯示 `label`；介面上點選兩個事件即可建立關係，不需手寫 JSON。`from`／`to` 只能指向**同一份文件內**的事件（跨文件的關係整合屬 Phase 2）。

## 8. `display` — 建議呈現方式

```json
"display": {
  "orientation": "horizontal",
  "defaultScale": "year",
  "range": { "start": "1986", "end": "2017" },
  "collapseGaps": true
}
```

只是「建議」，載入者可覆寫；使用者疊多個圖層時，以最外層的設定為準。`defaultScale`、`range`、`collapseGaps`（空白摺疊）都已實作，介面上也有對應的開關。

## 9. CSV／Google Sheet 匯入對照表

多數人會繼續用試算表填資料。匯入器必須容忍髒資料。

### 欄位對應（不分大小寫，支援別名）

| 標準欄位 | 接受的表頭 |
|---|---|
| `start` | `Start Date`, `Date`, `日期`, `開始日期` |
| （時間） | `Start Time`, `Time`, `時間` → 與日期合併成 TimePoint |
| `end` | `End Date`, `結束日期` |
| `title` | `Title`, `標題`, `事件` |
| `description` | `Description`, `說明`, `描述` |
| `location` | `Location`, `地點` |
| `track` | `Track`, `軸線`, `分類` |
| `tags` | `Tags`, `標籤`（逗號分隔） |
| `sources` | `Source`, `URL`, `來源` |

若無 `track` 欄，**試算表的每個分頁（sheet）視為一條軸線**——這正是你 Sample 檔案的既有結構（同婚立法／平權運動／食安事件）。

### 髒資料處理規則（**這些不是理論，全部來自你的真實資料**）

| 狀況 | 處理方式 |
|---|---|
| 整列空白 | 略過，不報錯 |
| 有日期無標題 | 略過，記入警告清單 |
| `Start Time` 是 `09:00-18:00` | 取 `09:00` 為 start，`18:00` 為 end（同日） |
| 結束日期寫「至今／迄今／持續中」 | 設為進行中事件（`ongoing: true`） |
| 日期寫成 `2017/02/20` 與 `2017/3/24` 混用 | 一律正規化 |
| 日期是 `2010年6月` | 解析為 `precision: "month"` |
| 完全無法解析的日期（如 `你好天`） | **不要靜默丟棄**。收進「待修正」清單，在介面上顯示給使用者處理 |
| 同日同標題重複列（如 2016/12/26 出現三次） | 匯入時標記為疑似重複，**由使用者決定是否合併**，程式不自動刪 |
| 欄位整體錯位的壞列 | 收進「待修正」清單 |

> **原則：匯入器永遠不靜默丟資料。** 所有無法處理的列都要回報給使用者。這是公民科技工具的基本尊重——資料是人辛苦整理的。

## 10. 版本演進規則

- `hackstory` 版本號採 `major.minor`
- **新增選填欄位** → minor +1，舊檔案仍可讀
- **移除或改變既有欄位語意** → major +1，必須提供轉換器
- 程式讀到不認識的欄位時**保留不動**（forward compatibility），存檔時原樣寫回
