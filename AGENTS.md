## 0. 我是誰、我的限制

我沒有程式基礎，沒有用過 GitHub。我看得懂中文說明、看得懂畫面上發生什麼事，但我**看不懂程式碼在幹嘛**。

因此請遵守：

1. **每一步都要能被我「用眼睛驗證」。** 不要一次寫五個檔案然後說「好了」。每完成一個小功能，告訴我怎麼打開來看、應該看到什麼。
2. **用中文解釋你做了什麼、為什麼。** 但不要教我語法，除非我問。
3. **每個里程碑結束時，主動提醒我存檔（commit），並幫我寫好 commit 訊息。**
4. **出錯時，先講「哪裡壞了、影響是什麼」，再講怎麼修。** 不要直接貼一堆錯誤訊息給我。
5. **不要自作主張擴大範圍。** 我沒要求的重構、沒要求的新功能、沒要求的「順便優化」，一律不要做。做之前先問。
6. 每次改動後，如果有任何指令要我在終端機執行，**完整寫出來，一行一行，附上執行後應該看到什麼**。

## 1. 這個專案在做什麼

**HackStory：多人協作、模組化的視覺化時間軸工具。**

源自 2016–2017 年的 g0v 專案（提案人：劉李俊達、吳翊瑋、黃嘉旭），當年因缺乏前端工程師而停滯。現以 Codex 重啟。

### 要解決的問題

1. **公共討論缺乏共同認知基礎。** 大家對同一議題的資訊落差極大，導致討論永遠停在「釐清基本事實」，無法推進。
2. **現有工具（TimelineJS、時間地圖）只呈現「個別事件」，不呈現「事件之間的因果與趨勢」，也沒有多軸對比與協作。**
3. **紙本限制。** 審議民主場合常用海報紙＋便利貼共建時間軸，難以保存、編輯、發佈。

### 目標使用者

- **一般公民**：面對龐雜資訊，想整理成看得懂的時間歷程
- **倡議團體**：釐清議題前因後果、做情境推演、對外說明
- **新媒體**：報導中的大事紀與脈絡整理

### 三個差異化定位（**這是專案的靈魂，任何設計決策不得違背**）

1. **多軸對比** — 同時呈現多條時間軸，看出彼此的互動
2. **可組合** — 一條時間軸是可攜檔案，能發佈、取用、與他人的疊加混合
3. **相對時間** — 不知道確切日期、只知道先後順序，也能建立時間軸

## 2. 非目標（Non-goals）— 明確不做的事

- ❌ 不做通用的專案管理／甘特圖工具
- ❌ 不做地圖工具（`location` 欄位預留了經緯度，但 MVP 不畫地圖）
- ❌ 不做社群功能（追蹤、按讚、動態牆）
- ❌ 不做行動版**編輯**（行動版只要能**看**得舒服）
- ❌ MVP 不做登入、不做資料庫、不做即時同步

## 3. 架構：六層，嚴格分離

```
6. ui/        頁面外殼、工具列、設定面板
5. share/     發佈到共用庫 / 取用他人時間軸   ← Phase 2
4. compose/   圖層管理器（疊加多份時間軸、配色、顯示隱藏、排序）
3. render/    視覺化引擎（橫軸/直軸、時間尺度、空白摺疊、事件繪製）
2. adapters/  匯入匯出（CSV / Google Sheet / JSON / 圖片）
1. core/      資料模型、SPEC 驗證、時間解析、相對時間求解
```

### 鐵律

- **每一層只能呼叫比它低的層。** `core` 不准 import 任何其他層。`render` 不准知道資料是從 CSV 還是資料庫來的。
- **`core` 不准碰畫面，`render` 不准碰資料來源。**
- 新增一種匯入格式 → 只動 `adapters/`
- 換一種畫法 → 只動 `render/`
- 接資料庫 → 只動 `share/`

> **這條鐵律是「模組化」的實質內容。** 每次你要寫程式前，先問自己：這段程式該住在哪一層？如果它需要同時知道兩層的事，代表設計錯了，停下來跟我討論。

## 4. 技術棧

| 項目 | 選擇 | 理由 |
|---|---|---|
| 建置工具 | Vite | 快、設定少 |
| 框架 | React + TypeScript | TypeScript 的型別檢查是我這種非工程師的安全網——寫錯會當場報錯，而不是上線後才炸 |
| 樣式 | Tailwind CSS | |
| 時間刻度數學 | `d3-scale` + `d3-time` | **只借用計算，不用它畫圖** |
| 繪製 | 自己用 SVG 畫 | 見下方說明 |
| CSV 解析 | `papaparse` | |
| 測試 | Vitest | `core/` 層必須有測試 |
| 部署 | GitHub Pages | 免費、靜態 |

### ⛔ 禁止使用 vis.js / vis-timeline

2017 年的原型用 vis.js，這是專案卡住的技術原因之一。我們要的功能——**相對時間、事件關係線、大量空白摺疊、事件面積與時長等比例**——每一項都違反 vis.js 的核心假設，繞過它的限制比從零寫還累。

**渲染層自己用 SVG 畫。** 不要引入任何現成的時間軸函式庫。

### 其他禁令

- ❌ 不要為了「更好」而換掉上表的任何選擇，除非先跟我討論
- ❌ 不要引入 UI 元件庫（Material UI 等），我們的畫面很特殊
- ❌ 不要在 MVP 階段引入任何後端、伺服器、資料庫

## 5. MVP 範圍與開發順序

**一次做一個里程碑。做完、我確認看得到、commit，才進下一個。**

### M0 — 專案骨架
- Vite + React + TS + Tailwind 起手
- 六層資料夾建好（空的也要建）
- 跑得起來，畫面顯示「HackStory」
- **驗收：** 我在瀏覽器打開 localhost，看到標題

### M1 — core：資料模型
- 依 `SPEC.md` 定義所有型別
- `.hst.json` 的驗證器（讀進來檢查格式對不對）
- 時間解析器：處理 `2017/5/24`、`2010年6月`、`09:00-18:00`、`2017/02/20`
- **`core` 層必須有測試。** SPEC 第 9 節列出的每一種髒資料，其表頭對應與分流邏輯屬「特定匯入格式」的知識，住在 `adapters/rows.ts`（見架構第 3 節鐵律），**由 adapters 的測試涵蓋**——不放 core
- **驗收：** 給我看測試全綠的畫面，並用中文列出「這些測試在測什麼」

### M2 — render：把一條軸畫出來
- 讀入 `examples/` 裡的範例檔，畫出一條橫向時間軸
- 點事件 = 圓點，區間事件 = 長條
- 滑鼠滾輪縮放、拖曳平移
- 日／週／月／年 尺度切換按鈕（像 Google 日曆）
- **驗收：** 我看到 1986–2017 的同婚時間軸

### M3 — 多軸與多圖層
- 同一份文件的多條 `track` 上下並排
- **同時載入多份 `.hst.json`，每份是一個圖層，各有顏色**
- 圖層面板：顯示／隱藏、調整順序、改配色
- **驗收：** 我把「同婚立法」和「平權運動」兩份檔案同時載入，看到它們並排對比

### M4 — adapters：匯入
- 上傳 CSV
- 貼上 Google Sheet 公開網址（用「發布到網路」的 CSV 連結，**不碰 Google API 授權**——這正是 2017 年翻車的地方）
- 匯入預覽畫面：顯示「成功 N 筆、警告 N 筆、無法解析 N 筆」，並讓我逐筆修正
- **驗收：** 我把 Drive 上那份 HackStory_Sample 貼進來，壞資料被列出來讓我處理，好資料畫成軸

### M5 — 匯出與分享
- 匯出 `.hst.json`
- 匯出 PNG / SVG 圖片
- 產生嵌入用的 iframe 程式碼
- **驗收：** 我下載檔案、下載圖片，並把時間軸嵌入一個測試網頁

### M6 — 上線
- 部署到 GitHub Pages
- **驗收：** 我把網址傳給朋友，他打得開
